## üìù Contribution Guidelines

### Before You Start
1. Check existing [issues](https://github.com/theclay1/crossover_learning_plan/issues) to avoid duplicates
2. For major changes, open an issue first to discuss
3. Keep contributions aligned with the project's goal: helping people prepare for Crossover

### Making Changes

#### For Small Changes (typos, links, etc.):
1. Click "Edit" directly on GitHub
2. Make your change
3. Submit a pull request with a clear description

#### For Larger Changes:
1. Fork the repository
2. Create a feature branch:
   ```bash
   git checkout -b feature/your-feature-name
   ```
3. Make your changes
4. Commit with clear messages:
   ```bash
   git commit -m "Add: New CCAT practice resource"
   ```
5. Push to your fork:
   ```bash
   git push origin feature/your-feature-name
   ```
6. Open a Pull Request

### Commit Message Format

Use clear, descriptive commit messages:

- **Add:** `Add new LeetCode problem patterns`
- **Fix:** `Fix broken link to HackerRank tutorial`
- **Update:** `Update CCAT preparation timeline`
- **Remove:** `Remove outdated resource`
- **Docs:** `Improve explanation of study tracker`

### Content Guidelines

#### ‚úÖ Do:
- Keep content free (or clearly mark paid resources as optional)
- Provide clear, actionable advice
- Include specific examples where helpful
- Maintain a supportive, encouraging tone
- Credit original sources appropriately
- Test links before submitting

#### ‚ùå Don't:
- Add affiliate links or paid promotions
- Include copyrighted content without permission
- Make unsupported claims about success rates
- Share confidential Crossover information
- Add offensive or discriminatory content

### Resource Quality Standards

When adding resources, ensure they are:
- **Free or have a substantial free tier**
- **High quality** and well-reviewed
- **Relevant** to Crossover preparation
- **Active** and maintained (not abandoned projects)
- **Accessible** (working links, no broken pages)

### Documentation Standards

- Use clear, simple language
- Break up long sections with headers
- Include examples where helpful
- Use markdown formatting consistently
- Check spelling and grammar

## üêõ Reporting Issues

### Bug Reports
Please include:
- Clear description of the problem
- Steps to reproduce (if applicable)
- Expected vs actual behavior
- Screenshots (if helpful)
- Your environment (browser, OS) if relevant

### Feature Requests
Please include:
- Clear description of the feature
- Why it would be helpful
- How it might work
- Any examples from other resources

### Broken Links
Please include:
- Which file contains the broken link
- The broken URL
- A working alternative (if you found one)

## üìã Pull Request Process

1. **Fork & Branch:** Create a feature branch from `main`
2. **Make Changes:** Edit files with your improvements
3. **Test:** Ensure all links work and formatting is correct
4. **Commit:** Use clear, descriptive commit messages
5. **Push:** Push changes to your fork
6. **PR:** Open a pull request with detailed description
7. **Review:** Respond to any feedback from maintainers
8. **Merge:** Once approved, changes will be merged!

### PR Description Template
```markdown
## Description
[Clearly describe what you changed and why]

## Type of Change
- [ ] Bug fix (broken link, typo, etc.)
- [ ] New feature (new resource, section, etc.)
- [ ] Documentation improvement
- [ ] Other (please describe)

## Checklist
- [ ] I've tested all links
- [ ] I've checked for typos
- [ ] I've followed the style guidelines
- [ ] My changes are clear and well-documented

## Additional Notes
[Any other context or screenshots]
```

## üîÑ Review Process

- Pull requests are typically reviewed within 3-7 days
- We may ask for changes or clarifications
- Be patient and responsive to feedback
- Once approved, changes are merged promptly
